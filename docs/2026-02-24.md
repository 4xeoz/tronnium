# Debounced Asset Position Updates

## Overview

This feature prevents excessive API calls when updating asset positions (e.g. during drag operations).

Instead of persisting every position change, updates are **debounced per asset**, meaning:

* The position is only saved after the user stops moving the asset.
* Each asset is debounced independently.
* Only the last position within the debounce window is persisted.

---

## Problem

During drag interactions, position updates can fire dozens of times per second.

Calling the mutation on every movement:

```ts
updatePositionMutation.mutate({ assetId, x, y })
```

Would:

* Spam the backend
* Cause unnecessary database writes
* Introduce potential race conditions
* Impact overall performance

---

## Solution

We use a **per-asset debounce mechanism** backed by `useRef` to store timeout handlers.

### Implementation

```ts
import { useRef, useEffect } from 'react'

const positionUpdateTimeouts = useRef<
  Record<string, ReturnType<typeof setTimeout>>
>({})

const debouncedUpdatePosition = (
  assetId: string,
  x: number,
  y: number
) => {
  // Clear any existing timeout for this asset
  if (positionUpdateTimeouts.current[assetId]) {
    clearTimeout(positionUpdateTimeouts.current[assetId])
  }

  // Schedule a new update
  positionUpdateTimeouts.current[assetId] = setTimeout(() => {
    updatePositionMutation.mutate({ assetId, x, y })
  }, 500)
}

// Cleanup on unmount
useEffect(() => {
  return () => {
    Object.values(positionUpdateTimeouts.current).forEach(clearTimeout)
  }
}, [])
```

---

## How It Works

1. Each `assetId` has its own timeout.
2. When a new position update comes in:

   * The previous timeout (if any) is cleared.
   * A new timeout is scheduled.
3. If no new updates occur within `500ms`, the mutation executes.
4. Only the final position is persisted.

---

## Example Timeline

If asset `A` is dragged continuously:

| Time  | Action          | Result                      |
| ----- | --------------- | --------------------------- |
| 0ms   | Move to (10,10) | Timeout scheduled           |
| 120ms | Move to (12,10) | Previous timeout cleared    |
| 300ms | Move to (20,10) | Previous timeout cleared    |
| 800ms | No movement     | Mutation fires with (20,10) |

Only one API call is made.

---

## Why `useRef`?

We use `useRef` because:

* It stores mutable values without triggering re-renders.
* It persists across renders.
* It is ideal for managing timers and external side effects.

Using `useState` here would cause unnecessary re-renders during drag events.

---

## Configuration

* **Debounce delay:** 500ms

  * Can be adjusted depending on UX needs.
  * Typical drag debounce range: 150–300ms.

---

## Benefits

* Reduces backend load
* Prevents race conditions
* Improves drag performance
* Ensures final position consistency

---

## Future Improvements

This logic can be extracted into a reusable custom hook such as:

```ts
useDebouncedPerKey()
```

Which can later be reused for resize, rotate, or other high-frequency updates.
